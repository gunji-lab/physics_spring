# physics_spring Googleログイン版 導入手順

## 現行版の扱い

現行の `physics_spring`、現行Web URL、現行GASデプロイは変更しない。新しい版は `google/` として別URLで試験公開する。

## 仕組み

- 大学Googleアカウントから、GASがサーバー側で学籍番号を取得する。
- メールアドレス全体は保存せず、`@` より前の学籍番号だけを記録する。
- ブラウザが申告した学籍番号は採用せず、署名付き認証情報をGASが検証して確定する。
- 認証情報は12時間で期限切れになる。
- Stage一覧には「✓ クリア」「○ 未クリア」を表示する。

## 1. 新版を別URLへ配置する

このフォルダを、現行版とは別の `google/` というパスで公開する。現行版を上書きしない。

公開URLの例：

```text
https://（現在のサイト）/physics_spring/google
```

## 2. GASコードを準備する

現在利用中のスプレッドシートに紐づいたApps Scriptで、現行コードをバックアップする。そのうえで、新しいコードバージョンとして `gas/physics_trainer_gas_dashboard_exclude_test_v2.gs` の内容を使用する。

古いWebアプリのデプロイは削除・更新しない。デプロイは特定バージョンを保持するため、古いURLは従来の動作を続ける。

GASコード内の次を、新版の公開URLへ変更する。

```javascript
const NEW_TRAINER_BASE_URL = "https://（現在のサイト）/physics_spring/google";
```

## 3. 新しいデプロイを2つ作る

同じGASコードから、用途の異なるWebアプリを2つ新規作成する。

### A. 認証用

- 実行するユーザー：アクセスしているユーザー
- アクセスできるユーザー：大学組織内のユーザー
- 用途：大学メールの確認と署名付き認証情報の発行

### B. 記録・進捗API用

- 実行するユーザー：自分
- アクセスできるユーザー：全員
- 用途：署名を検証した結果だけを記録・返却

APIは外部公開されるが、正しい署名がない提出は保存しない。

## 4. 新版HTMLへURLを設定する

次の2ファイルにあるプレースホルダーを、手順3で発行されたURLへ置き換える。

- `common/trainer_common_v2.js`
- `common/trainer_portal_v3.js`

```javascript
__AUTH_DEPLOYMENT_URL__  // AのURL
__API_DEPLOYMENT_URL__   // BのURL
```

## 5. 少人数で試験する

1. 教員アカウントでトップページを開く。
2. 学生1〜3名に新版URLを案内する。
3. 学籍番号の手入力なしでStageを開始できることを確認する。
4. 70%以上の結果送信後、Stage一覧へ戻りチェックが付くことを確認する。
5. スプレッドシートにメールアドレスではなく学籍番号だけが記録されることを確認する。
6. 現行URLが従来どおり利用できることを確認する。

## 注意

認証用デプロイとAPI用デプロイを取り違えない。試験完了までは現行版を学生へ案内し続け、新版は限定した学生だけに案内する。
